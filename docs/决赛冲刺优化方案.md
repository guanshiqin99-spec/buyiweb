## v2 修订摘要（2026-07-23 核查结论）

经逐项核对代码库，原方案引用的代码点（`selectedItem`/`mapResults`/`startQuiz`/`scoreAnswers`/`learnStats`/`askStream`/CSS 变量/路由等）**全部真实存在**，方案总体可执行。但有以下修订：

1. **新增 P0-0（最高优先级）**：线上站点 API 链路已断。`buyi-dictionary-vue/functions/api/[[path]].js` 指向已失效的 trycloudflare 临时隧道（实测 `https://buyi-dictionary.pages.dev/api/health` 返回 Cloudflare error 1003），且该代理函数**不转发请求体和 Authorization 头、不透传 SSE 流** —— 意味着线上登录、注册、收藏、学习记录、AI 问答全部不可用。不修这一项，后面所有优化在评委面前都是零分。
2. **修正红线自相矛盾**：第四章第 2 条"禁止修改 api.js"与 P0-2 的"api.js 追加 agentApi"冲突。结论：**取消"api.js 追加 agentApi"子项**（现有 AI 调用本就走独立的 `agentStream.js`，不经过 axios；新增 `generateStream` 直接放在 `agentStream.js` 即可，少改一个文件）。
3. **修正 P0-2 三处实现风险**（详见 P0-2 修订标注）：AI 出题从"调 5 次"改为"一次生成 5 题"；关联推荐从 `watch(selectedItem)` 自动触发改为按钮手动触发；新增 `generate` 端点必须同步在 `main.ts` 追加一行限流路径（原方案漏了，`main.ts` 现有限流只覆盖 `/api/miniapp/agent/ask`）。
4. **红线例外澄清**：第四章第 15 条"禁止删除任何现有文件"适用于**源代码**；仓库中的归档残留（`temp_extract/`、`.git_backup/` 等）按《docs/handover/01-冗余文件清理清单.md》单独处理，不受此条约束。
5. **P1-1 锚点跳转写法修正**：本项目路由是 hash 模式（`createWebHashHistory`），跨页锚点必须用对象形式 `:to="{ path: '/culture', hash: '#pattern-exhibit' }"`（Culture.vue:35 已有同款先例），不能写字符串 `to="/culture#pattern-exhibit"`。

## 第一章：不可逾越的边界红线

> ⛔ **以下规则在整个优化过程中始终生效，任何阶段不得违反。**

### 1.1 范畴边界定义

| 范畴 | 目录范围 | 可做的事 | 禁止做的事 |
|------|----------|----------|------------|
| **网页端** | `buyi-dictionary-vue/src/**` | 新增组件、新增工具函数、在现有页面追加模板片段 | 修改现有组件的核心业务逻辑、修改路由守卫、修改 axios 拦截器 |
| **小程序端** | `BuyiDictionaryApp-main/BuyiDictionaryApp-main/pages/**` 和 `utils/**` | 本次优化**不涉及**小程序端改动 | 任何改动 |
| **后端** | `BuyiDictionaryApp-main/BuyiDictionaryApp-main/backend/src/**` | 仅在 `miniapp-agent` 模块内新增 1 个 controller 方法 | 修改任何现有 controller/service/entity/guard、修改 app.module.ts 的 imports 顺序 |
| **数据库** | MySQL / SQLite | **不新增表、不修改表结构、不新增字段** | 任何 schema 变更 |
| **Service Worker** | `buyi-dictionary-vue/public/learning-reminder-sw.js` | 在文件末尾追加 fetch 事件监听 | 修改现有 push/periodicsync/notificationclick 事件处理 |

### 1.2 代码修改红线

1. **只加不改**：所有改动以"新增文件"或"在现有文件末尾追加"形式进行。不修改任何现有函数的签名、参数、返回值。
2. **新增组件必须自包含**：新组件不修改任何现有组件的 props/emits/slots。新组件引用现有 store 和 API 时，只读不写。
3. **降级优先**：任何新功能的失败不得阻塞现有功能。所有新 API 调用必须有 try-catch，失败时静默降级。
4. **不引入新依赖**：不安装任何新的 npm 包。所有新功能用 Vue 3 + 原生 CSS + Canvas API + SVG 实现。
5. **不碰数据库**：不执行任何 migration、不修改 entity 文件、不新增 entity。

### 1.3 部署安全保证

- 所有新增前端文件在 `npm run build` 后自动打包，无需修改构建配置
- 后端新增端点需要重新 `npm run build` 并重启 PM2，但现有端点不受影响
- 即使新增功能代码存在 bug，最坏情况是该功能不显示或报错，不影响现有页面正常加载
- 所有新增组件使用 `v-if` 守卫，数据加载失败时不渲染

## 第二章：各优化项的范畴归属与实现方向

### P0-0：修复线上 API 链路（v2 新增，最高优先级）

**目标**：让 `https://buyi-dictionary.pages.dev` 的后端功能真正可用。当前实测：首页可打开（静态资源正常），但 `/api/*` 全部失败。

#### 问题定位（已核实）

| # | 问题 | 文件:行号 | 影响 |
|---|------|-----------|------|
| 1 | `BACKEND_URL` 指向 trycloudflare 临时隧道（已失效） | `buyi-dictionary-vue/functions/api/[[path]].js:1` | 所有 API 请求 502/1003 |
| 2 | 代理 fetch 未转发请求 body | 同文件 `fetch(targetUrl, {...})` 处 | 即使隧道恢复，登录/注册/收藏等所有 POST/PUT 也会失败 |
| 3 | 代理未转发 `Authorization` 头（只发 `accept`） | 同上 | 所有鉴权接口返回 401 |
| 4 | `await response.text()` 缓冲整个响应 | 同上 | SSE 流式 AI 变成"卡住后一次性吐出"，且长对话可能超时 |
| 5 | `functions/api/health.js` 硬编码旧 IP `http://39.96.81.132` | `buyi-dictionary-vue/functions/api/health.js` | 与主代理不一致，健康检查结果误导排查 |

#### 实现方向

1. **确定稳定后端地址**（前置决策，需人工确认）：三选一 —— ① Sealos 部署（`sealos-deploy.yml` 已备好，得到 `*.bja.sealos.run` 已备案 HTTPS 域名，推荐）；② 阿里云 ECS + **命名** Cloudflare Tunnel（`cloudflared tunnel create`，域名固定，区别于会变址的 quick tunnel）；③ 后端换任何带稳定 HTTPS 域名的托管。**禁止再用 trycloudflare quick tunnel**——它重启就换地址，评审期内必断。
2. **重写 `functions/api/[[path]].js` 为透明代理**（整文件重写，约 30 行）：
   - `BACKEND_URL` 改从 Pages 环境变量 `context.env.BACKEND_URL` 读取，代码中只留兜底注释——以后换后端只需在 Cloudflare 控制台改变量并重新部署，不改代码；
   - 转发原始 method、`Authorization`/`Content-Type`/`Accept` 头、原始 `request.body`；
   - 直接 `return new Response(upstream.body, { status, headers })` 透传流（SSE 天然可用），不再 `await text()`；
   - 保留 OPTIONS 预检处理。
3. **删除 `functions/api/health.js`**（`[[path]].js` 已覆盖 `/api/health`），`functions/ping.js` 可留作 Pages Function 存活探针。
4. **验收**：本地 `npx wrangler pages dev dist` 验证后再部署；线上验证 `curl https://buyi-dictionary.pages.dev/api/health` 返回 `{"status":"ok"}`，再走一遍登录→收藏→AI 问答。

#### 操作边界

- 只动 `buyi-dictionary-vue/functions/` 下文件，**不碰任何 Vue 源码、后端源码**
- 后端 `CORS_ORIGIN` 需包含 `https://buyi-dictionary.pages.dev`（改后端 .env，非代码）
- 风险极低：代理函数独立于 SPA 构建产物，改坏也只影响 API 转发，静态页面不受影响

#### 具体实现方向

**新建 `ShareCard.vue`**：
- 暴露 `generate(options)` 方法，返回 `Promise<Blob>`

  - 背景：靛蓝渐变（`#1b3a5c` → `#0d2538`）

## 第三章：实施顺序与验收标准

### 实施顺序（v2 修订）

```
P0-0 修复线上API链路（最先，独立于一切前端优化）
        ↓
P0-1 权威溯源体系 → P0-3 词汇分享卡片 → P0-2 AI能力深度化
                                            ↓
                                    P1-1 内容关联导航
                                            ↓
                                    P1-2 学习数据可视化
                                            ↓
                                    P1-3 个性化学习建议
                                            ↓
                                    P2（时间充裕时选做）
```

**推荐顺序理由**：
1. **P0-0 必须最先做**：线上 API 现在是断的，这决定评委能否体验任何后端功能；且它只动 `functions/` 两个文件，与其余优化零耦合
2. P0-1 最安全（纯展示组件），先做可验证部署流程无变化
3. P0-3 其次（纯前端 Canvas），无后端依赖
4. P0-2 最后（涉及后端），前两项验证无误后再动后端；注意 P0-2 的线上体验依赖 P0-0 完成（SSE 需经修好的透传代理）
5. P1 项均无后端依赖，顺序灵活

**验证方式**：改动验证优先本地进行——双击根目录 `启动竞赛包.bat`（该脚本用相对路径，可直接用）同时起前后端，浏览器访问 `http://127.0.0.1:5173`。注意 `启动前端.bat`/`启动后端.bat`/`导入示例数据.bat` 三个脚本内硬编码了旧路径 `D:\BuyiDictionaryWeb\`，在当前仓库位置会直接报错（修复项见 handover 清单）。线上验证只在 P0-0 部署后做一轮，避免频繁访问服务器。
