# 重大漏洞与修复清单

> 生成时间：2026-07-23。按严重程度排序。原则：赛前只做小而稳的修复，每项都给出最小改动方案。

## 致命（评审前必须处理）

### V1. 线上 API 链路完全断裂
- **现象（已实测）**：`https://buyi-dictionary.pages.dev` 首页可开，但 `GET /api/health` 返回 Cloudflare error 1003。线上登录、搜索（后端源）、收藏、学习记录、AI 问答全部不可用。
- **根因**：`buyi-dictionary-vue/functions/api/[[path]].js:1` 的 `BACKEND_URL` 指向 trycloudflare **临时**隧道（进程重启即失效）。且该代理函数还有三个独立 bug：
  1. `fetch(targetUrl, {...})` 未传 `body` → 所有 POST/PUT 请求体丢失（登录必失败）
  2. 只转发 `accept` 头，不转发 `Authorization` → 所有鉴权接口 401
  3. `await response.text()` → SSE 流被整体缓冲，AI 流式变一次性输出
- **修复**：见《决赛冲刺优化方案.md》P0-0。核心是①后端落到稳定 HTTPS 地址（推荐 Sealos，配置已备好）；②重写 `[[path]].js` 为透传代理（转发 method/关键头/body，`return new Response(upstream.body, ...)`）；③ `BACKEND_URL` 走 Pages 环境变量。
- **风险**：低。只动 functions 目录，不碰 SPA。

- `backend/src/config/app.config.ts:88`：`process.env.DB_SYNCHRONIZE ?? 'true'`。TypeORM 自动同步在生产可能因实体改动静默改表/丢数据。另外 `database.config.ts:46` 的 migrations 数组只注册了 5 个迁移中的前 3 个（RBAC 与 song-duration 两个迁移没挂上）——说明线上 schema 实际靠 synchronize 维持。
- **修复（赛前最稳做法）**：不改代码，仅在生产 `.env` 显式写 `DB_SYNCHRONIZE=false`（schema 已建好后关闭）。迁移数组的不一致赛后再理。

## 低 / 观察项

- **Pages Function 代理会绕过后端 CORS**：代理把 `access-control-allow-origin: *` 写死，等于对外开放了一个任意来源可用的后端转发口。透传重写时保留 OPTIONS 处理但把 `*` 收敛为 `https://buyi-dictionary.pages.dev` 更稳妥（同源场景其实可以完全不加 CORS 头）。
- **前端路由守卫仅是 UX 屏障**：后端各 miniapp 端点已用 `MiniappJwtGuard` 真实校验（已核实 agent 控制器整体挂 guard），无越权问题，仅提示不要误以为前端守卫是安全边界。
- **`npm run lint`（backend）只是 `tsc --noEmit`**：不是真 lint，别被脚本名误导。
- 后端整体安全基线良好：helmet、全局 ValidationPipe（whitelist+forbidNonWhitelisted）、登录/上传/AI 限流、bcrypt、双 token 会话、JWT_SECRET 无代码级 fallback（缺失即启动失败）。
