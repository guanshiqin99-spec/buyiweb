```
┌─────────────── 用户访问层 ──────────────────┐
│   Web 评委浏览器              微信小程序      │
│   https://buyi-dictionary     wx.cloud.      │
│       .pages.dev              callFunction   │
└──────────┬────────────────────────┬──────────┘
           │ HTTPS                  │ 微信自有通道
           │                        ▼
           │              ┌─────────────────────┐
           │              │  云函数 apiProxy    │
           │              │  (微信服务器)       │
           │              └──────────┬──────────┘
           │                         │ HTTP 直连 IP
           ▼                         │ http://39.96.81.132:80/api
┌──────────────────────┐             │
│ Cloudflare Pages     │             │
│ (前端静态资源+HTTPS) │             │

           │       .asia/api         │
           ▼                         │
┌──────────────────────────┐         │
│ Cloudflare Named Tunnel  │         │
│ api.buyitech.asia        │         │
│ (稳定 HTTPS 域名)        │         │
└──────────┬───────────────┘         │
           │                         │
           └────────────┬────────────┘
                        ▼
         ┌──────────────────────────────┐
         │ 阿里云北京 ECS 39.96.81.132  │
         │ (Ubuntu 22.04, 2c4g)         │
         ├──────────────────────────────┤

         │ Nginx :80 → 反代 :3000       │

         │ CORS_ORIGIN 含 pages.dev     │

```

### 1.2 核心优势

| 问题 | 解决方案 |
|------|---------|
| Web 端需要稳定 HTTPS | **Cloudflare Pages + Named Tunnel**：固定域名 `api.buyitech.asia`，服务器重启不失效 |
| 小程序需要备案域名 | **云函数代理**：小程序调云函数，云函数直连后端 IP，绕开白名单限制 |
| 后端需要稳定运行 | **阿里云北京 ECS**：国内服务器稳定、低延迟 |
| 成本控制 | Cloudflare 全免费 + 阿里云试用额度 + 云函数免费额度 |

| 小程序 | 云函数代理 → `http://39.96.81.132:80/api`（走 Nginx 反代到 3000） | ❌ 无需备案 |

## 三、Cloudflare Named Tunnel 配置

> **目标**：把 `https://api.buyitech.asia` 稳定映射到本机 `localhost:3000`，服务器重启后自动恢复。
>
> **状态自检**：若你已在服务器上跑过 `cloudflared tunnel login` 和 `cloudflared tunnel create buyi-backend`，可跳过 3.2、3.3，直接从 3.4 开始。

### 3.1 安装 cloudflared

```bash
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 -o /usr/local/bin/cloudflared
chmod +x /usr/local/bin/cloudflared

cloudflared --version
```

### 3.2 登录 Cloudflare

```bash
cloudflared tunnel login
```

命令会输出一个 URL，复制到浏览器打开，选择 `buyitech.asia` 区域授权。授权完成后本地生成 `~/.cloudflared/cert.pem`。

### 3.3 创建 Tunnel 并绑定 DNS

```bash
# 创建隧道（隧道名 buyi-backend，Tunnel ID 会自动生成）
cloudflared tunnel create buyi-backend

# 把 api.buyitech.asia 绑定到该隧道（自动写入 Cloudflare DNS CNAME）
cloudflared tunnel route dns buyi-backend api.buyitech.asia
```

执行完后 Cloudflare DNS 会自动出现：
- 类型 `CNAME`，名称 `api`，内容 `<TUNNEL_ID>.cfargotunnel.com`，代理状态橙云。

### 3.4 写入隧道配置文件

```bash
cat > ~/.cloudflared/config.yml << 'EOF'
tunnel: <TUNNEL_ID>
credentials-file: /root/.cloudflared/<TUNNEL_ID>.json

ingress:
  - hostname: api.buyitech.asia
    service: http://localhost:3000
  - service: http_status:404
EOF
```

> 把 `<TUNNEL_ID>` 替换为 `cloudflared tunnel create` 输出的 UUID，并确认 `~/.cloudflared/` 下存在 `<TUNNEL_ID>.json` 凭证文件。

### 3.5 安装为 systemd 守护进程（关键：开机自启）

```bash
cloudflared service install
systemctl enable cloudflared
systemctl start cloudflared

# 验证
systemctl status cloudflared
# 应为 active (running)
```

> **不要用 `cloudflared tunnel run` 前台启动**：服务器重启后会失效，导致 `api.buyitech.asia` 访问不通。

### 3.6 验证 Tunnel

```bash
# 服务器本机
curl https://api.buyitech.asia/api/health
# 应返回: {"status":"ok",...}
```

## 五、小程序云函数代理配置

### 5.1 核心原理

```
小程序客户端
    │ wx.cloud.callFunction({ name: 'apiProxy', data: { path, method, data } })
    ▼
云函数 apiProxy（运行在微信服务器）
    │ http.request({ hostname: '39.96.81.132', port: 3000, path, method })
    ▼
阿里云后端
    │
    ▼
返回数据 → 云函数 → 小程序客户端
```

**关键点**：
- 云函数运行在微信服务器，**不受小程序白名单限制**
- 可以直接访问 HTTP IP、未备案域名
- 无需在微信公众平台配置服务器域名

### 5.2 代码已就绪

| 文件 | 作用 |
|------|------|
| [`cloudfunctions/apiProxy/index.js`](file:///d:/BuyiDictionaryWeb/BuyiDictionaryApp-main/BuyiDictionaryApp-main/cloudfunctions/apiProxy/index.js) | 云函数代理逻辑，直连阿里云 IP |
| [`utils/runtime-config.js`](file:///d:/BuyiDictionaryWeb/BuyiDictionaryApp-main/BuyiDictionaryApp-main/utils/runtime-config.js) | 云函数配置（envId、functionName） |
| [`app.js`](file:///d:/BuyiDictionaryWeb/BuyiDictionaryApp-main/BuyiDictionaryApp-main/app.js) | requestApi 方法，优先调用云函数 |

### 5.3 配置后端 IP

`cloudfunctions/apiProxy/index.js` 第 9 行已配置为（走 Nginx 80 端口反代到 NestJS 3000）：

```javascript
// 阿里云北京服务器公网 IP，走 Nginx :80 反代到 NestJS :3000
const BACKEND_BASE = 'http://39.96.81.132:80/api';
```

> **关于 BACKEND_BASE 的策略**：
> - 小程序走云函数时，**直连 IP** 最稳定，不依赖 Cloudflare Tunnel
> - 端口用 `:80`（Nginx 入口）而非 `:3000`（NestJS 直连）：Nginx 负责静态资源、 uploads 媒体路径转发与统一日志，且防火墙只开放 80/443，3000 不对外暴露
> - 若你希望小程序也走 Tunnel 域名，可改为 `https://api.buyitech.asia/api`，但要承担 Tunnel 故障时小程序也挂的风险
> - 当前推荐：IP + 80 端口直连 Nginx（与 Web 通道解耦）

## 附录：常见问题

### Q1: `https://api.buyitech.asia/api/health` 访问不通？

**排查顺序**：

1. 服务器本机 `curl http://localhost:3000/api/health` 是否 OK
   - 失败 → `pm2 restart buyi-backend`
2. `systemctl status cloudflared` 是否 active
   - 失败 → `systemctl restart cloudflared`
3. `~/.cloudflared/config.yml` 中 `tunnel:` 和 `credentials-file:` 路径是否正确
4. Cloudflare DNS 中 `api` 记录是否为 CNAME 指向 `<TUNNEL_ID>.cfargotunnel.com`，代理状态橙云

### Q2: Web 端 F12 看到 CORS 错误？

**原因**：后端 `CORS_ORIGIN` 未包含 `https://buyi-dictionary.pages.dev`。

**解决**：
1. 编辑 `/opt/buyi-dictionary/backend/.env`，确保 `CORS_ORIGIN=https://buyi-dictionary.pages.dev`
2. `pm2 restart buyi-backend`

### Q3: 服务器重启后 `api.buyitech.asia` 失效？

**原因**：cloudflared 没装成 systemd 服务，仅前台运行。

**解决**：
```bash
cloudflared service install
systemctl enable cloudflared
systemctl start cloudflared
```

### Q4: 云函数调用失败，提示 FUNCTION_NOT_FOUND？

**原因**：云函数未上传或环境 ID 不匹配。

**解决**：
1. 确认已在微信开发者工具中右键 `cloudfunctions` → 选择正确的云环境
2. 重新上传云函数：右键 `apiProxy` → "上传并部署"
3. 等待 1-2 分钟后再测试

### Q5: 云函数调用成功，但返回 500 错误？

**原因**：后端 IP 配置错误或后端服务未启动。

**解决**：
1. 检查 `cloudfunctions/apiProxy/index.js` 中的 `BACKEND_BASE` 是否为 `http://39.96.81.132:80/api`
2. 在服务器上执行 `pm2 logs buyi-backend` 查看后端日志
3. 测试 `curl http://localhost:3000/api/health`

### Q6: 小程序正式版无法调用云函数？

**原因**：正式版环境 ID 与代码中配置不一致。

**解决**：
1. 在微信公众平台查看正式版的云开发环境 ID
2. 更新代码中的 `envId` 并重新上传小程序代码

### Q7: 音频文件 404？

**原因**：`MEDIA_PUBLIC_BASE_URL` 与实际音频路径不一致。

**解决**：
1. 后端 `.env` 中 `MEDIA_PUBLIC_BASE_URL=https://api.buyitech.asia/uploads`
2. 服务器上确认 `/opt/buyi-dictionary/backend/uploads/buyi_audio/*.MP3` 存在
3. `pm2 restart buyi-backend`
